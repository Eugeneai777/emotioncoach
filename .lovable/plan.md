## 方案：稳定小程序/微信/手机/电脑端分享海报生成流程

目标：用户点击“生成/分享海报”后，不再依赖远程上传或 AI 背景生成完成才显示图片；先稳定生成本地海报预览，再后台升级为可长按保存的 HTTPS 图片，保证各端流程一致。

### 一、核心流程改造

将分享海报生成统一改为“两阶段”：

```text
点击生成分享海报
  -> 立即在前端渲染海报 canvas
  -> 转成本地 Blob 图片
  -> 立刻打开海报预览页，用户能看到图
  -> 后台异步上传图片
  -> 上传成功后自动替换成 HTTPS 图片，方便小程序/微信长按保存
  -> 上传失败也不影响当前预览，提示用户可重试或重新生成
```

这样解决“小程序端要试几次才生成、生成后才能保存”的不稳定体验。

### 二、统一修复分享入口

会统一处理以下入口，避免不同页面行为不一致：

- 海报中心快速模式
- 海报中心专家模式
- `PosterGenerator` 的“一键分享 / 下载海报”
- 通用 `ShareImagePreview` 预览保存组件
- 通用 `oneClickShare` / `shareUtils` 分享工具

### 三、小程序端专门增强

针对小程序 WebView 性能和资源限制，会做以下优化：

1. 降低小程序/微信环境默认 canvas 渲染压力，减少内存失败。
2. 首次生成失败时自动降级重试一次：降低 scale、跳过部分等待、避免直接报错。
3. 预览页打开后增加“图片准备中/可长按保存”的状态区分。
4. 远程 HTTPS 图未准备好时，先显示本地图；远程图准备好后自动替换。
5. 关闭预览时只回收本地 Blob URL，避免错误回收远程 URL。

### 四、修复图片格式不一致问题

当前部分逻辑实际输出 JPEG，但文件名和 File 类型仍按 PNG 处理，微信/小程序端可能出现预览或保存异常。

会统一：

- canvas 转 Blob 的 MIME 类型
- 分享 File 的 MIME 类型
- 下载文件后缀
- 预览保存文件名

确保生成、预览、保存链路格式一致。

### 五、AI 背景生成与分享海报解耦

如果用户选择 AI 背景，AI 背景生成可能因为后端耗时、网络或额度原因失败。这个失败不应该导致最终分享海报无法生成。

会调整为：

- AI 背景失败时，自动保留当前纯色/渐变背景。
- 用户仍可继续生成普通分享海报。
- 错误提示改成中文可理解提示，而不是直接“生成失败”。

### 六、用户提示优化

优化提示文案，让用户知道当前状态：

- “正在生成海报...”
- “海报已生成，正在准备高清保存图...”
- “高清图已准备好，可长按保存”
- “网络较慢，当前图片可预览；如无法保存请点重新生成”

### 涉及文件

- `src/utils/shareCardConfig.ts`
- `src/utils/oneClickShare.ts`
- `src/utils/shareUtils.ts`
- `src/components/ui/share-image-preview.tsx`
- `src/components/poster/PosterGenerator.tsx`
- `src/pages/PosterCenter.tsx`
- 必要时微调 `supabase/functions/generate-poster-image/index.ts` 的错误提示和返回信息

### 不做的事情

- 不改数据库结构。
- 不改变现有海报视觉设计。
- 不影响普通浏览器/电脑端原有保存方式。
- 不把 AI 图片生成放到前端，避免暴露密钥。

### 验收标准

完成后应满足：

1. 小程序端点击生成后能尽快看到海报预览。
2. 上传失败不导致整张海报生成失败。
3. 微信、小程序、安卓、iOS、电脑端都走稳定一致的预览保存流程。
4. 重试次数明显减少，不需要用户反复点击多次才成功。
5. AI 背景失败时仍能生成基础分享海报。
